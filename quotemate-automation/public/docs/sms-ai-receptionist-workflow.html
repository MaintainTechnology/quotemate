1. Executive Summary

QuoteMax's SMS AI Receptionist is a channel-agnostic quoting pipeline that turns an inbound customer text into a grounded, auto-sent Good/Better/Best quote (or a $99 inspection fallback). It is anchored by three Next.js 16 App Router routes running on next/server after() with maxDuration=300: app/api/sms/inbound/route.ts (the 2767-line conversational front door), app/api/intake/structure/route.ts (the channel-agnostic structurer), and app/api/estimate/draft/route.ts (the orchestrator that persists, prices and delivers). Inbound Twilio webhooks are signature-validated, tenant-routed, deduped, locked, and fast-acked with empty TwiML; all heavy work runs in the background. A Claude Sonnet 4.6 dialog/slot/intent loop maintains a slot store (sms_conversations.conversation_state jsonb) until a deterministic readiness gate (evaluateQuoteReadiness) is satisfied at action='finish', which hands off to intake structuring. Intake structuring runs Claude Opus 4.8 (cascading to 4.7 then Sonnet 4.6) vision over transcript + photos to emit one intakes row, embedded via Voyage voyage-3-large (1024-dim). Estimation (Opus 4.8) drafts G/B/B using TOOL-CALLING ONLY for prices, anchored by RAG over the tenant's own past quotes. The hard backstop is lib/estimate/validate.ts: every line-item price must derive from pricing_book + shared_* + tenant_custom_assemblies + tenant_material_catalogue scoped to intake.trade; ANY ungrounded line downgrades the entire quote to the $99 inspection route. On success, the quote persists to quotes.good/better/best jsonb, Stripe test-mode Checkout sessions are created (30% per-tier deposit, or single $99 inspection), the send/hold gate shouldHoldForReview() (default auto_send = Path B) decides whether to auto-send, and delivery goes SMS-first with WhatsApp fallback plus a best-effort Gotenberg PDF MMS, while the tradie is notified. Money is stored ex-GST and displayed inc-GST (×1.10). Multi-tenant isolation is app-layer tenant_id filtering on a service-role Supabase client (RLS enabled-but-policy-less). External services: Anthropic (Claude), Voyage (embeddings + rerank), Cohere (alternate rerank), Google Gemini (preview/sample images), Twilio (SMS/MMS/WhatsApp), Stripe (test-mode Checkout), Gotenberg (PDF). Stripe Connect Express schema exists but is unwired; the payments table has 0 rows.

3. Phase-by-Phase Walkthrough

1

Inbound SMS & AI Receptionist Dialog

A signature-validated Twilio webhook fast-acks with empty TwiML, then a Claude Sonnet 4.6 slot-extract + dialog loop in after() converses with the customer one turn at a time, persisting a jsonb slot store.

DATA IN Twilio x-www-form-urlencoded inbound: From, To, Body, MessageSid, NumMedia, MediaUrl0..N, header x-twilio-signature.
DATA OUT An updated sms_conversations.conversation_state jsonb (merged slots), an appended sms_messages inbound row, a stored set of MMS photo paths, and a TurnDecision with action and a sent customer reply.

Every inbound text lands as a form-encoded POST on app/api/sms/inbound/route.ts (the 2767-line front door). The route first HMAC-validates the Twilio signature via validateTwilioSignature (lib/sms/twilio-validator.ts), reconstructing the URL from x-forwarded-host/proto, and 403s on mismatch; it then resolves the owning tradie by destination number with tenantByDestinationSms, resolves/stubs the customer with findOrCreateCustomer (stamping tenant_id), and dedupes on twilio_message_sid using an app-layer maybeSingle plus a DB partial unique index (migration 004) so Twilio webhook retries short-circuit to an empty 200. Any MMS media (MediaUrl0..N) is pulled via Basic-auth GET to Twilio and stored to the Supabase intake-photos bucket by extractAndStoreMmsPhotos, the inbound row is appended to sms_messages, and a per-conversation lock is claimed atomically (processing_until, migration 007) so concurrent webhooks for the same customer coalesce and bail. Critically, the route then returns an EMPTY TwiML <Response/> (so Twilio never auto-replies) and runs ALL heavy work in next/server after() with maxDuration=300 — fast-ack stays under 500ms while a worst-case finish turn can run ~200s. Inside after() a 1500ms debounce coalesces rapid-fire inbounds, then extractSlots (Claude Sonnet 4.6, generateText + manual JSON parse to dodge Anthropic's 'schema too complex' tool_use error) performs turn-by-turn NLU and merges results into the sms_conversations.conversation_state jsonb (slots + source attribution), and decideNextTurn (Claude Sonnet 4.6, generateObject + Zod TurnDecisionSchema) returns one action ∈ ask|finish|escalate_inspection|end_conversation plus reply_to_send. Deterministic scrubs and guards police model drift — buildGpoInspectionOverride stops false-positive power_points→inspection escalations, Rule 5/6 forces the universal name+suburb must-asks, and scrubbers strip voice wording and smart quotes — and all three LLM steps were upgraded from Haiku 4.5 to Sonnet 4.6 on 2026-05-14 for Rule 0 ('read before asking') precision at ~5x cost. A separate maybeHandleTradieRegistration branch runs only when no tenant owns the destination number, classifying intent (regex + Sonnet 4.6) and minting a JOIN-code signup intent rather than entering the customer-quote loop.

Readiness Gate & Hand-off to Intake

On action='finish' a deterministic readiness gate confirms every price-critical fact is captured before the route fires a retrying POST into the intake pipeline.

DATA IN A TurnDecision with action='finish', the accumulated conversation_state slots, known first_name/suburb, full message history and the tenant's in-scope services.
DATA OUT Either a downgraded 'ask' follow-up SMS (gate failed), or a fired POST to /api/intake/structure carrying {conversationId, sourceChannel:'sms'} (gate passed).

The conversation only escapes the dialog loop when decideNextTurn returns action='finish'. At that boundary the route calls evaluateQuoteReadiness (lib/sms/quote-readiness.ts), a purely deterministic gate that the model cannot override: it checks the universal must-ask fields (name / suburb / job_type) plus the per-job REQUIRED_BY_JOB facts — e.g. downlights need count, room, ceiling_type, replace_or_new and colour; hot_water needs energy_source, litres and room — and if anything is missing it DOWNGRADES the action from 'finish' back to 'ask' and emits a focused follow-up, preventing a low-confidence quote. A stack of idempotency guards also runs here so reuse and continuation never double-fire the pipeline: quoteAlreadyDrafted (lib/sms/quote-already-drafted.ts), isQuoteInflight (lib/sms/inflight.ts) and a hasExistingIntake check; notably 'done' is deliberately NOT treated as drafted (a fix for the end_conversation ghosting bug). A WP9 product-choice interlock can also hold the quote open while the customer picks between two catalogue products (product_choice jsonb). Only when the gate passes AND the quote is not already drafted, not WP9-held and not inflight does the route fire withRetry (3 attempts, 2s/4s backoff) a POST to /api/intake/structure {conversationId, sourceChannel:'sms'} — the hand-off into the quote pipeline; shouldRetry is false on abort/timeout because intake/structure may still complete server-side, avoiding a duplicate pipeline run. If all retries are exhausted, the route sends buildQuoteFailureSms and reopens the conversation so the customer is never left silent.

Intake Structuring & Embedding

The channel-agnostic structurer stitches the dialog into a transcript, runs Claude Opus 4.8 vision over text + photos to emit one structured intakes row, embeds it with Voyage voyage-3-large, then quality-gates the hand-off to estimation.

DATA IN A POST {conversationId, sourceChannel:'sms'} (or {callId}); resolves to a stitched Customer:/Agent: transcript, de-duped photo URLs/paths, a tradeHint and resolved tenant_id.
DATA OUT One inserted intakes row (structured jsonb + 1024-dim embedding + photos + trade), and either a fired POST to /api/estimate/draft (usable) or a recovery SMS (empty).

app/api/intake/structure/route.ts (POST, maxDuration=300) is the single channel-agnostic entry point, accepting either {callId} (voice) or {conversationId, sourceChannel:'sms'}. For SMS it loads the sms_conversations row plus all sms_messages, stitches them into a Customer:/Agent: transcript, prepends the dialog agent's assumptions_made, and aggregates photo URLs/paths from BOTH inbound MMS (sms_messages.photo_urls) and upload-link uploads (sms_conversations.photo_urls), de-duplicated via a Set. It then calls structureIntake(transcript, photoUrls, tradeHint, modelId) in lib/intake/structure.ts through a three-model cascade (INTAKE_MODEL_CASCADE = ['claude-opus-4-8','claude-opus-4-7','claude-sonnet-4-6']) wrapped in withRetry; structuring uses Anthropic generateObject (Vercel AI SDK v6) with a Zod schema, and photos ARE used for vision as {type:'image', image:url} content parts. A key design choice: trade is NEVER classified by the LLM — StructureSchema is IntakeSchema.omit({trade}) (with scope extended by one REQUIRED requested_specs_json string to work around Anthropic's 24-optional-field cap), and trade is computed post-hoc by deriveTradeFromJobType(object.job_type) against a hardcoded PLUMBING_JOB_TYPES set (everything else, including 'other'/'renovation', defaults to electrical). Confidence (LOW/MEDIUM/HIGH) and a required confidence_reason are set by Opus per an explicit rubric in the system prompt. The structured object is then embedded by embedIntake (lib/intake/embed.ts) into a 1024-dim Voyage voyage-3-large vector — but over a compact summary string (trade/job_type/item_count/is_new_install/indoor_outdoor/risks), NOT the raw transcript — falling back to a deterministic FNV-hash stub if VOYAGE_API_KEY is unset. Customer memory is resolved and used to backfill ONLY blank intake fields, an optional gas-HWS inspection override may fire (off by default), and a single intakes row is inserted with scope/access/property/risks/timing/caller jsonb, the embedding, photo_paths and tenant_id. Finally evaluateIntakeQuality (lib/intake/quality.ts) decides 'usable' vs 'empty': 'empty' reopens the conversation and sends a focused recovery SMS, while 'usable' dispatches a withRetry POST to /api/estimate/draft inside after().

RAG Retrieval of Similar Past Quotes

The new intake's embedding drives a pgvector match_intakes search pre-filtered by job_type, survivors are hydrated with their winning quotes and Voyage-reranked to the top 3, then prepended to the Opus estimation prompt's user message.

DATA IN The new intake's 1024-dim Voyage embedding plus its job_type (used as the exact pre-filter).
DATA OUT A formatted 'SIMILAR PAST QUOTES' text block (top 3 reranked comparable jobs) prepended to the Opus estimation user message.

At estimate time, fetchSimilarPastQuotesContext (lib/estimate/rag.ts) anchors the Opus draft to the tradie's own past work. It calls the Postgres function match_intakes(query_embedding vector(1024), match_count=20, job_type_filter=intake.job_type), which runs pgvector cosine search (1 - (embedding <=> query)) PRE-FILTERED by exact job_type. Survivors above MIN_SIMILARITY=0.40 are hydrated with their winning quote, explicitly EXCLUDING inspection-required and null-tier drafts so only real priced quotes inform the new one. The candidates are then cross-encoder re-ranked by Voyage rerank-2.5 (lib/estimate/rerank.ts), keeping the top FINAL_CONTEXT_COUNT=3 with rerank score >= MIN_RERANK_SCORE=0.30; on rerank failure or when disabled, getReranker() returns null and the system degrades gracefully to cosine ordering (a Cohere reranker exists as an alternative via chainWithFallback). The surviving top 3 are formatted into a 'SIMILAR PAST QUOTES' text block that is prepended to the Opus USER message in lib/estimate/run.ts — deliberately not the system message, so Anthropic ephemeral prompt caching on the cacheable system prompt is preserved while the per-tenant RAG context varies. Two important grounded caveats: the embedding column was migrated from vector(1536) to vector(1024) in migration 057 (init.sql and several rag.ts header comments still stale-read '1536-dim'), and when VOYAGE_API_KEY is unset the upstream stub embedding makes retrieval non-semantic — RAG silently degrades but the pipeline still completes.

Estimation Drafting (Tool-Calling Only)

runEstimation calls Claude Opus 4.8 ONCE with a trade-specific prompt and four tools, where the model is forbidden from free-forming any dollar amount and must derive every price from a tool result.

DATA IN The structured intakes row, the resolved tenant+trade pricing_book, and the prepended RAG 'SIMILAR PAST QUOTES' context block.
DATA OUT A parsed Good/Better/Best draft JSON (scope, assumptions, risk_flags, tier line_items with source tags) OR a self-reported needs_inspection signal.

runEstimation (lib/estimate/run.ts) turns the structured intake plus the tenant-scoped pricing_book into a Good/Better/Best draft via a single Claude generateText call (Vercel AI SDK v6) with stopWhen: stepCountIs(10) so the model can iterate tool calls. The default model id is 'claude-opus-4-8' (Opus 4.8); the calling route app/api/estimate/draft/route.ts drives a cascade Opus 4.8 -> Opus 4.7 -> Sonnet 4.6 via withRetry. The system prompt is trade-specific and data-driven (electrical NSW/NECA via electrical-prompt.ts, plumbing QLD/QBCC via plumbing-prompt.ts, selected by systemPrompt() in prompt.ts with normaliseTrade defaulting empty trade to electrical), with placeholder rates computed by buildEstimatorContext (prompt-context.ts), and it carries Anthropic ephemeral prompt caching on the system message. The hard rule taught throughout: EVERY line_item.unit_price_ex_gst must come from a tool result — lookupAssembly, lookupMaterial, applyMarkup or pricing_book rates — never free-form. The prompt also fixes the calculation order: labour_hours = quantity x assembly.default_labour_hours (NOT 1hr/unit), labour_total = labour_hours x hourly_rate, material_total = quantity x unit price, material_marked_up = applyMarkup(material_total), line_total = labour + marked-up material, with apprentice_rate for high-volume GOOD labour and senior_rate for complex BEST labour plus risk_buffer_pct and after-hours multipliers when triggered. The Opus output is a JSON object with scope_of_works, assumptions[], risk_flags[] and good/better/best tiers of line_items (each {description, quantity, unit, unit_price_ex_gst, total_ex_gst, source:'material:<id>'|'assembly:<id>'}), extracted by parseJsonFromText (strips fences then string-aware brace-walks). If Opus self-reports needs_inspection=true, run.ts returns immediately and the route nulls the tiers and applies the $99 fee. Otherwise a sequence of post-processors runs — optional deterministic-BOM rebuild (flag DETERMINISTIC_BOM), applyMinLabourFloor topping each tier's labour up to pricing_book.min_labour_hours so small jobs are not bounced to inspection, a price-bands recipe merge, and optional KB verification (flag KB_VERIFY_ESTIMATES) — all best-effort so any failure keeps the Opus draft unchanged.

Catalog & Pricing Resolution

The four pricing tools resolve prices from four trade- and tenant-scoped data layers, UNIONing tenant-owned catalogues ahead of the shared library and reranking to the best SKU before applying the tenant's markup.

DATA IN Tool calls from Opus (query + trade + property filters) plus the intake's tenant_id and trade.
DATA OUT Resolved priced rows (ex-GST) with source ids and a markup-applied sell price feeding the draft's line_items.

Every dollar the model uses is resolved through makeTools(tenantId) (lib/estimate/tools.ts) reading four data layers, all scoped by trade (electrical|plumbing) and tenant_id. Layer 0 is the per-(tenant,trade) pricing_book row supplying hourly_rate, apprentice_rate/senior_rate, default_markup_pct, risk_buffer_pct, min_labour_hours, call_out_minimum and GST/licence; it is resolved by resolvePricingBookForIntake (lib/estimate/pricing-book.ts), a HARD tenant-ownership gate — if no book exists or the loaded book's tenant_id/trade mismatches the intake, it returns ok:false and the route synthesizes an inspection-only draft, NEVER borrowing another tradie's rates (the WP1 rule). lookupAssembly UNIONs the global shared_assemblies with the tenant's tenant_custom_assemblies (custom-first), located by name-ILIKE with synonym expansion; lookupMaterial UNIONs shared_materials with the tenant's tenant_material_catalogue (tenant-first). Both apply property filters (color_temp, dimmable, smart, weatherproof, supplied_by), fetch up to 12 rows, then Voyage-rerank to top-5 so the best SKU is row #1. Each row's price comes from a fixed column — default_unit_price_ex_gst (assemblies + shared_materials), unit_price_ex_gst (tenant tradie-supply), or customer_supply_price_ex_gst (install-only); customer-supply mode (applyCustomerSupplyMode) swaps in install-only pricing and drops rows lacking that price so the customer is never double-billed. applyMarkup multiplies basePrice by (1 + pct/100) at the tenant's default_markup_pct (defaulting 28). Supporting tables include shared_assembly_bom / tenant_assembly_bom (decomposing an assembly into material-category x quantity for the flag-gated deterministic path), tenant_assembly_overrides, tenant_service_offerings (the Services on/off toggle), tenant_material_preferences (soft brand hint) and tenant_tier_ladder (explicit G/B/B ladder). Two grounded subtleties: price_recipes is NOT a table — recipes live in a price_recipe jsonb column on shared_assemblies + tenant_custom_assemblies (migration 074); and supplier_catalogue is a browse-only vendor library NEVER read by the estimator or validator (tradies copy SKUs from it into their own catalogue at their own price).

Grounding Validation & Routing Decision

validateQuoteGrounding re-derives the allowed price universe and rejects any unmatched line — collapsing the whole quote to a $99 inspection — then decideRouting records a label and shouldHoldForReview sets the real send-vs-hold gate.

DATA IN The post-processed Good/Better/Best draft (line_items with source tags) plus the candidate price set re-derived from pricing_book + shared/tenant catalogues scoped to trade.
DATA OUT Either a grounded G/B/B draft with a review hold/send decision, or a fully downgraded inspection-required draft (tiers nulled, $99 forced, [grounding] risk flags).

validateQuoteGrounding (lib/estimate/validate.ts) is the hard backstop after drafting. buildCandidatePrices / loadCandidatePrices re-derive the accepted price universe independently from the model: every pricing_book rate and every shared/tenant catalogue row scoped to intake.trade is expanded into accepted markup variants {0, default-5pp, default, default+5pp} with a +/-$0.50 tolerance, labour must equal hourly/apprentice/senior (or after-hours x multiplier when tagged), and callout must equal call_out_minimum. Each line item must match a candidate by BOTH price AND category, and strict UUID grounding (R-4) requires a source 'material:<id>'/'assembly:<id>' to resolve to that exact loaded row at a valid markup. If ANY line fails grounding — or labour falls below min_labour_hours, or a duplicate catalogue row appears twice in one tier (D-1) — the ENTIRE quote is downgraded to inspection-required: good/better/best are FORCED to null, needs_inspection=true, the rejected lines are appended to risk_flags as [grounding] entries, and the route forces the $99 site-visit total even if Opus emitted indicative numbers. The validator deliberately omits enabled/active filters (M-6) so a SKU disabled seconds after Opus grounded on it does not retroactively dump the quote. On success, deterministic reconcile backstops (lib/estimate/reconcile.ts) polish arithmetic, collapse duplicate tiers and flag quantities, and a spec guard (default 'enforce') blocks products contradicting the customer's agreed specs. 6. Materials Catalog & Pricing Database — Deep Dive

QuoteMax prices a job by combining four trade- and tenant-scoped data layers, and the LLM never free-forms a number: every dollar comes from a tool result that reads a fixed price column, and a deterministic grounding validator re-derives the allowed price universe and rejects anything outside it. Follow ONE material — say a customer's downlight — from catalogue row to priced line item: (0) the tenant's pricing_book row supplies hourly_rate, default_markup_pct, min_labour_hours, GST and call_out_minimum; (1) Opus calls lookupMaterial('downlight', trade='electrical'); (2) the tool UNIONs the tenant's tenant_material_catalogue (active, trade-scoped) ahead of shared_materials, reranks to top-5, and aliases the chosen row's unit_price_ex_gst to default_unit_price_ex_gst; (3) Opus calls applyMarkup(rawPrice, default_markup_pct) → sell = raw×(1+28/100); (4) it adds a labour line = install_hours × hourly_rate; (5) applyMinLabourFloor tops the tier's labour up to min_labour_hours; (6) the line lands in the tier with source='material:<uuid>'; (7) validateQuoteGrounding re-derives that row's accepted markup variants {0,23,28,33}% ±$0.50 and confirms the price + the 'downlight' category match (strict-UUID path if a uuid source is present); (8) on pass the tier persists ex-GST and is displayed inc-GST (×1.10). Any single ungrounded line downgrades the WHOLE quote to a $99 inspection.

The price formula

PER MATERIAL/ASSEMBLY LINE (ex-GST): raw_price = catalogue row price column (default_unit_price_ex_gst, or aliased unit_price_ex_gst / customer_supply_price_ex_gst) material_marked_up = round2( raw_price × (1 + pricing_book.default_markup_pct / 100) ) // electrical 28%, plumbing 15-20% material_total = quantity × material_marked_up PER LABOUR LINE (ex-GST): labour_hours = quantity × assembly.default_labour_hours // NOT 1hr/unit labour_total = labour_hours × pricing_book.hourly_rate // electrical ~$110, plumbing ~$120; apprentice_rate (GOOD) / senior_rate (BEST); ×after_hours_multiplier (1.5) when tagged CALL-OUT (when triggered): callout_line = pricing_book.call_out_minimum (unit='each') MIN-LABOUR FLOOR: if Σ(tier 'hr' lines) < pricing_book.min_labour_hours (2.0 elec / 1.5 plumb) → add deficit @ hourly_rate RISK BUFFER: pricing_book.risk_buffer_pct (default 15) EXISTS but is NOT applied — configured-but-unused in the pricer TIER SUBTOTAL (ex-GST) = Σ(material_total) + Σ(labour_total) + callout (+ recipe-band extras) CUSTOMER TOTAL: subtotal_ex_gst + GST, GST = gst_registered ? round2(subtotal × 0.10) : 0, total_inc_gst = subtotal + GST DEPOSIT (per tier): tierIncGst = subtotal_ex_gst × 1.10; deposit = round(tierIncGst × 30 / 100) // depositPct hardcoded 30 INSPECTION ROUTE (fixed): total_inc_gst = $99 (INSPECTION_TOTAL_INC_GST), GST = 99/11, subtotal = 99 - 99/11; Stripe unit_amount = INSPECTION_FEE_AUD_CENTS = 9900
GST, markup & labour
GST: all prices stored and computed ex-GST; inc-GST is display-only (×1.10). GST = gst_registered ? round2(subtotal × 0.10) : 0; quotes carries gst + total_inc_gst columns. MARKUP: material/assembly sell = raw × (1+default_markup_pct/100) (electrical ~28%, plumbing 15-20%); applyMarkup default 28% is a safety net only; the validator accepts a ±5pp band plus 0% raw (relaxed 2026-05-13 to stop AU plumbing 20%-rounding bouncing a 15%-configured book to inspection). LABOUR: hourly_rate is the base; apprentice_rate for high-volume GOOD-tier labour, senior_rate for complex BEST-tier labour, hourly × after_hours_multiplier (default 1.5) for after-hours; min_labour_hours (2.0 elec / 1.5 plumb) is the per-tier floor topped up at hourly_rate before grounding. risk_buffer_pct (default 15) is configured but NOT applied in any pricing math.
Grounding validator contract
CONTRACT (lib/estimate/validate.ts:197-562): every line_item.unit_price_ex_gst in every non-null tier must derive from the loaded candidate set scoped to intake.trade. Accepted sources: pricing_book labour rates (hourly_rate / apprentice_rate / senior_rate, or hourly × after_hours_multiplier when the source is tagged after_hours, matched ±0.5); call_out_minimum on a unit='each' line (or source='callout'); a material/assembly price matching a candidate at a markup variant {0%, default-5pp, default, default+5pp} within PRICE_TOLERANCE ±$0.50. Two matching paths: (a) STRICT-UUID when source='material:<id>'/'assembly:<id>' — must match THAT exact row's variants, no category fallback (a fabricated/wrong/cross-trade id fails); (b) LOOSE — price match AND keyword-category overlap (categorise() + explicit row.category, mig 029). Also enforced: per-tier summed 'hr' labour >= min_labour_hours (lineIndex -1), and the D-1 duplicate guard (same catalogue row twice in one tier fails, preventing raw + marked-up double-billing). The candidate load deliberately OMITS the active filter on tenant rows (M-6) so a row deactivated seconds after Opus grounded on it still validates. RETURN: {valid:true} | {valid:false, failures:[{tier, lineIndex, description, unit, unit_price_ex_gst, expected}]}. ANY failure → run.ts sets good/better/best=null, needs_inspection=true, inspection_reason='Pricing not yet available — N line item(s) failed grounding check…', and the route forces the $99 inspection total.

11. Glossary

Path B (auto-send)
The live behaviour: every clean drafted quote auto-sends to the customer and the tradie is notified after the fact. The default review_policy='auto_send'; supersedes the original 'no auto-send in v1' decision.
Grounding validator
lib/estimate/validate.ts — the hard 4th-line-of-defence that requires every line price to derive from pricing_book + shared_* + tenant_custom_assemblies + tenant_material_catalogue scoped to intake.trade; any failure downgrades the whole quote to $99 inspection.
Inspection fallback
The safe failure mode: tiers nulled, needs_inspection=true, a fixed $99 inc-GST total and a single Stripe site-visit session instead of a priced quote.
Tool-calling only
The money-path rule that the LLM may produce a dollar amount ONLY from a tool result (lookupAssembly/lookupMaterial/applyMarkup) or pricing_book rates — never free-form.
routing_decision vs shouldHoldForReview
routing_decision (decideRouting) is a RECORDED legacy string with no live effect on dispatch; the operative send/hold gate is shouldHoldForReview reading pricing_book.review_policy.
conversation_state
The sms_conversations jsonb slot store (slots + source attribution) maintained by extractSlots; replaced wholesale each turn, so durable pins (followup_quote, product_choice, roofing_state) live in separate columns.
Readiness gate
evaluateQuoteReadiness — deterministic 'enough info to quote?' check on action='finish' (universal + per-job REQUIRED_BY_JOB facts + custom clarifying_questions) that downgrades finish→ask when facts are missing.
WP9 product-choice
A flag-gated mid-chat interlock that offers 2 tenant_material_catalogue products, holds the quote open (product_choice status='pending') until the customer picks, so the priced quote reflects the chosen SKU.
Min-labour floor
applyMinLabourFloor tops each tier's labour up to pricing_book.min_labour_hours (2.0 elec / 1.5 plumb) at hourly_rate BEFORE grounding, so small jobs aren't bounced to inspection.
Markup drift band
The validator accepts {0%, default-5pp, default, default+5pp} (±$0.50) so Opus rounding (e.g. 20% on a 15% book) still grounds without dumping to inspection.
Strict-UUID grounding
When a line's source is 'material:<id>'/'assembly:<id>', the price must match that exact row's markup variants with no category fallback; fabricated/wrong/cross-trade ids fail.
Customer-supply mode
applyCustomerSupplyMode swaps in customer_supply_price_ex_gst (install-only) when the customer supplies the product, and drops rows lacking that price (H-1).
easy-5
The five (electrical) / six (plumbing) job types eligible for the photo-request trigger and the deterministic happy path.
after()
next/server background execution: webhooks fast-ack within milliseconds then run all heavy LLM/DB/dispatch work in after() under maxDuration=300.
