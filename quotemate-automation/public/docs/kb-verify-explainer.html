01 · The one-liner

A second opinion on price, before the quote is built

When a customer texts in, the Estimate Engine drafts a Good / Better / Best quote. The drafted prices can drift. The MT-QM-PRICING-KB verification node asks the authoritative file store — which holds an indexed copy of every Supabase pricing and materials row — for the real price of each line, then verifies, supplements, and corrects the draft so the number the customer receives is accurate.

Verify

Each priced line is matched to the KB's authoritative figure and marked confirmed when it agrees.

Supplement

Where the KB disagrees, a [kb-verify] risk flag is attached so the tradie sees exactly what to check before sending.

Correct

In apply mode the mismatched unit price is overwritten with the KB figure — then re-grounded against the database.

03 · Zoom in

Inside the Estimate Engine — the exact seam

The node runs before the grounding validator on purpose. A KB price rewrite in apply mode must still pass grounding — if the KB's figure isn't itself traceable to the database, the existing validator catches it and the whole quote self-downgrades to the $99 inspection route. The KB feeds the validator; it never replaces it.

04 · The node itself

How the node works, step by step

Five deterministic steps wrap one KB search. Every step has a safe exit — the moment anything is missing, the node returns null and the draft is byte-identical to the KB-off path.

05 · The dial

One flag, three settings

Behaviour is controlled by the KB_VERIFY_ESTIMATES environment variable. It ships off, so production is byte-identical until you turn it up.

off · default

Unset / 0 / false. The node returns immediately. Zero behaviour change, zero KB calls, zero latency.

shadow · observe

1 / true / shadow. Verifies and attaches [kb-verify] risk flags for the tradie — but never changes a customer-facing price. The safe way to watch it work on real traffic.

apply · correct

apply. Also overwrites a mismatched unit price with the KB figure, then hands the corrected draft to the grounding validator.

06 · Per-line outcome

Three verdicts per line item

The reconciler token-matches each line description to a KB finding and compares prices within a ±10% tolerance (markups round; the answer paraphrases).

✓ Confirmed

The KB price is within tolerance of the drafted price. No action — the line is independently vouched for.

≠ Mismatch

The KB price differs by more than 10%. A correction is recorded and a risk flag is raised. In apply mode the price is overwritten.

? Uncovered

No confident KB match for this line. Left exactly as drafted — the node never guesses a price it didn't find.

What the tradie sees on a mismatch

[kb-verify] better: "Double GPO power point" drafted at $60.00 but MT-QM-PRICING-KB shows $45.00 (-25% off) — verify before sending.

07 · Why it can't hurt you

Accurate, never reckless

It feeds the backstop, never bypasses it

Because the node runs before the grounding validator, a KB correction still has to trace to pricing_book + shared_* + tenant assemblies. An ungrounded correction simply routes the whole quote to the $99 inspection — the same safe failure mode every money-path step uses.

It degrades to a no-op

KB disabled, unreachable, low-confidence, or returns no parseable price → the node returns null and the draft is byte-identical to today. The KB can never block or break an estimate.

It never free-forms money

The reconciler only ever copies a price the KB cited. It cannot invent a number — same rule the rest of the engine follows.

It is provably tested

18 unit tests cover the mode gate, query builder, price parser, reconciler, apply step, and the I/O wrapper across shadow / apply / off / KB-error / no-match. The full suite stays green.
