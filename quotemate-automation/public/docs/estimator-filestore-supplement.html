01 · The one-liner

A second pass on the take-off, before it's saved

The Electrical Estimator reads an uploaded plan PDF with Claude vision and counts every symbol into a quantity take-off. Vision counting is imperfect — dense fields and tiny legends produce gaps and miscounts. The file-store supplement node spins up a temporary Gemini File Search index from that same PDF, asks it for the quantities the plan's own schedules state, and uses the answers to verify, correct, and fill the take-off — then tears the index down. The store exists only for the length of one extraction.

Verify

An uncertain count that the plan's schedule text confirms is promoted to high confidence — the tradie has one less line to second-guess.

Correct

When the schedule states a different number, the count is overwritten with the document's figure and stamped [file-store] with the before→after.

Fill gaps

A legend symbol the vision pass missed entirely is added when the schedule gives it an explicit quantity — nothing slips through uncounted.

02 · Altitude

Where it sits in the estimator flow

The node is a sub-step inside the extract route — it runs after the Claude take-off and before the extraction is saved, so the supplemented numbers are what get persisted and shown. It is opt-in behind ESTIMATOR_FILESTORE_SUPPLEMENT_ENABLED; off by default, the route behaves exactly as before.

Default-off, fail-open

With the flag off — or the service unreachable, or the env unset — the node is skipped and the estimate proceeds on the original Claude take-off. Supplementation can only ever improve the result or be a no-op; it can never break the estimate.

03 · The ephemeral lifecycle

Create → upload → search → merge → delete

Five steps. The temporary store is alive only between step ① and step ⑤. Step ⑤ runs in a finally block, so the store is deleted whether the run succeeds, errors, or times out.

04 · Internals

Inside the node — three pure steps + one orchestrator

The logic is split the way the rest of lib/estimation is: small pure functions that are trivial to unit-test, wrapped by one thin orchestrator that does the I/O. Pure core, thin shell.

buildSupplementQueries(parsed)
pure · returns a bounded list of targeted queries

Targets the items worth re-checking first — every low then medium confidence count — plus any legend symbol that was never counted (a likely miss). High-confidence items are left alone. Capped (default 12) so cost stays bounded no matter how big the plan is.

↓per query
client.search(store, query)
io · POST /v1/search → { answer, citations[] }

Each query hits the file store. We keep only citations[].snippet — the raw passages retrieved from the plan's own schedules — and deliberately throw the answer field away (see §07).

↓evidence
findQuantityInSnippets(needles, snippets)
pure · returns a number | null

Reads an explicit quantity out of the passages — "Double GPO: 42", "42 × GPO", "GPO qty 42", "Downlight = 18". Returns null on no match or conflicting numbers, and ignores needles shorter than two characters so a stray "E" never matches.

↓apply
mergeSupplement(parsed, evidence)
pure · returns { parsed, changes[] }

Applies the evidence: correct a differing count, raise confidence on a confirmed one, or add a missed legend item — each recorded as a change with provenance and stamped on the item's note. Evidence with no explicit number changes nothing.

↓finally
client.deleteStore(store)
io · DELETE /v1/stores/:id?force=true

Runs in the orchestrator's finally, so the temporary store is removed on every path. A failed teardown is swallowed (the store is uniquely named and the upstream sweeps orphans) — it never surfaces as an estimate error.

05 · What the merge does

Four outcomes — three changes and a deliberate no-op

For each piece of evidence, mergeSupplement takes exactly one of these paths. The fourth — doing nothing — is the most important: the node never invents a number.

count_corrected

The schedule states a different number. The count is overwritten, confidence set to high, and the item note stamped [file-store] corrected 40→42 from plan schedule.

confidence_raised

The schedule confirms the existing count. The number is unchanged; confidence is promoted to high so the tradie can skip re-checking that line.

gap_filled

A legend symbol the visual pass never counted has an explicit quantity in the text. A new item is added at medium confidence, flagged as "added from plan schedule".

no change

The evidence carries no clear number, or conflicting numbers. The item is left exactly as the vision pass produced it. No fabrication, ever.

// mergeSupplement — the correction path (paraphrased) const qty = findQuantityInSnippets([item.type, item.symbol], ev.snippets) if (qty === null) continue // no clear evidence → no change if (qty !== item.count) { changes.push({ kind: 'count_corrected', before: item.count, after: qty }) item.count = qty item.confidence = 'high' item.note = stampNote(item.note, `corrected ${item.count}→${qty} from plan schedule`) }
06 · Guarantees

Why it's safe to put near the numbers

The node touches a tradie-facing quantity take-off, so every design choice is defensive. Six guarantees, each backed by a test.

Always ephemeral

The store is deleted in a finally block — even when upload or search throws. Tests assert the delete fires on the error paths and that no delete is attempted if the store was never created.

Never fabricates

A count only changes when a snippet states an explicit number. Ambiguous or numberless evidence leaves the extraction byte-for-byte unchanged.

Snippets, not the answer

Only raw retrieved passages drive merges; the synthesised answer is discarded. A test feeds answer:"999" + snippet "…: 42" and asserts the result is 42.

Fail-open

Unconfigured, unreachable, or any thrown error → the original Claude take-off passes through. The estimate never depends on the file store being up.

Flag-gated

Off unless ESTIMATOR_FILESTORE_SUPPLEMENT_ENABLED is true. Shipping the code does not change behaviour until it is switched on.

No new storage

Raw PDF bytes are not persisted anywhere new — the temporary index is the only external copy, and it is torn down. The api key and file bytes are never logged.

It enriches the count — it does not price it

Supplementation only adjusts the extracted take-off. The downstream pricing step is still deterministic and grounded against tenant_custom_assemblies + shared_assemblies + pricing_book; the node neither sets nor bypasses a single price.

07 · The snippet rule

Why we trust the citations, not the answer

The mt-filestore-kb service injects a fixed signage-compliance system persona into /v1/search — it was built first for a different product. That makes its synthesised answer unreliable for electrical content: it can editorialise, refuse, or speak in the wrong domain. The citations, however, are just verbatim passages pulled from the uploaded plan — domain-neutral and trustworthy.

// search response — we keep snippets and drop the answer { "answer": "As a signage-compliance engine…", // ← discarded (wrong persona) "citations": [ { "snippet": "POWER SCHEDULE — Double GPO: 42 total" }, // ← trusted { "snippet": "Exit sign quantity 3" } ] }

One env var to graduate this

If the file store later exposes a neutral or per-request system prompt, the node can begin trusting answer as a cross-check — but until then, snippets are the single source of evidence, by design.

09 · Verification

27 tests, no network

Every test runs on injected fakes — no live service, no model call — so the suite is fast and deterministic. The behaviours that matter are pinned down:

  • Cleanup is guaranteed: the store is deleted even when search or upload throws; no delete when the store was never created.
  • Snippets win: given a misleading answer and a correct snippet, the merge uses the snippet.
  • No fabrication: numberless / conflicting evidence leaves the take-off untouched.
  • Correct / confirm / fill: each merge outcome is asserted, including provenance stamping.
  • Graceful degrade: a null client (unconfigured) passes the original extraction straight through, and never creates a store when there's nothing to verify.
  • Client wire-format: x-api-key header, multipart field file, ?force=true on delete, and FileStoreError carrying the HTTP status.
