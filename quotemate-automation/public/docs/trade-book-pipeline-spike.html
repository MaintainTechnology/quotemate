01Target document format

The pipeline works only on documents with the right shape. Trying to extract from invoices ("replaced 6 LED downlights, $300") gives us nothing useful. The right input is the document a tradie hands to a new apprentice — usually called a work-instruction manual, an internal pricing guide, or a standard operating procedure (SOP) book.

Good input — what we want
• Work-instruction manuals (per-job procedure + timing)
• Internal pricing guides (per-service labour breakdowns)
• Apprentice training docs (assumptions + escalation rules)
• Supplier price catalogues (parts + brands)
• SOPs ("how we price a downlight install")
Bad input — won't work
• Customer-facing invoices ("replaced 6 LED downlights $300")
• Quotes already sent (output, not source)
• Photos of completed work (no pricing data)
• Generic compliance docs (standards, not pricing)

Accepted formats

Format Supported by scanner Notes
PDF (text-native)✓ BestCleanest extraction. Most modern work-instruction manuals.
PDF (scanned image)✓ Via OCROlder books — Gemini handles OCR transparently but accuracy depends on scan quality.
DOCX / DOC✓ Convert to PDF firstEasy pre-processing step in the upload flow.
XLSX (spreadsheet)~ LimitedTables extract OK; structured rows can be parsed directly without Gemini if the layout is consistent.
Image (JPG/PNG)~ Page-by-page onlySingle-page snapshots OK; multi-page books are clumsier than PDF.
Decision needed from Jon

Do we limit the initial pilot to PDF-only? Recommendation: yes — covers 80% of trade books, no upload-flow complexity. Add DOCX/XLSX in v2 once we know the extraction prompt is stable.

02Extraction prompt schema

The prompt we send to mt-filestore-kb against the indexed document. Returns structured JSON that maps 1:1 to a row in shared_assemblies (or tenant_custom_assemblies for tenant-specific uploads). The shape includes all the columns we've added through migration 067 — so the extracted output can populate row_assumptions, clarifying_questions, and inspection_triggers directly.

The prompt we send

// Sent to mt-filestore-kb POST /v1/search against the indexed document "For every install or repair service described in this document, extract the following structured fields. Return a JSON array, one object per service. Cite the page/section where you found each one." // Required schema per service: { "trade": "electrical | plumbing | carpentry | ...", "name": "Short descriptive name, e.g. 'Install LED downlight (new install)'", "description": "1-2 sentence summary of the work performed", "category": "downlight | gpo | smoke_alarm | hot_water | ...", "default_unit": "each | hr | lm", "default_unit_price_ex_gst": 35.00, "default_labour_hours": 1.75, "default_exclusions": "What's explicitly NOT included", "clarifying_questions": [ "Question 1 that affects the price", "Question 2..." ], "row_assumptions": { "switch_within_metres": 5, "max_storeys": 1, "roof_access_required": true }, "inspection_triggers": [ "raked ceiling", "multi-storey", "asbestos" ], "properties": { "weatherproof": false, "new_install": true }, "always_inspection": false, "materials": [ { "name": "LED downlight 9W warm white", "brand": "Clipsal", "unit": "each", "unit_price_ex_gst": 28.00 } ], "source_citation": "Page 12, 'Standard Downlight Procedure'" }

Schema aligns with what we already have

Every field maps to an existing column in shared_assemblies + shared_materials (after migrations 029, 032/033, 041, 065, 067). The only NEW DB change needed for this pipeline is adding a source_ref column on import_staged_rows so the operator review UI can show citations alongside each extracted row.

Operator review then becomes a quick scan

04UI tweaks needed

The admin loader review tool at /admin/loader already handles staging + edit + approve. Three small additions to make it work cleanly for trade-book uploads:

1
"Upload trade book PDF" button + source-document selector
Replaces the existing "Upload CSV" button with a dropdown: "Upload CSV" or "Upload trade book PDF". PDF path triggers the scanner + extraction prompt instead of CSV parsing.
~2 hours frontend work
2
Source-citation column in the staging table
Each row in the review list shows the citation back to the PDF page + section. Click → opens the original PDF at that location for verification. Requires a new source_ref column on import_staged_rows (one-line migration).
~3 hours full-stack work
3
"Bulk approve" with per-row override
Right now the loader is one-row-at-a-time. For a trade book with 50+ extracted rows, a "select all → approve" with inline edit beats clicking 50 times. The data-model already supports it; just the UI control to add.
~2 hours frontend
