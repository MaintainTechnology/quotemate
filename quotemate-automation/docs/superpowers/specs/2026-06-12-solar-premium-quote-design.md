# Solar Premium Quote — design spec

**Date:** 2026-06-12
**Status:** Approved (design review with owner, 2026-06-12)
**Owner decisions baked in:** Pylon "light now, import later" · deterministic overlays + Gemini hero · optional bill field · pure-SVG build (Approach A)

---

## 1. Goal

Raise the solar estimate's customer-facing output (the `/q/solar/[token]` page and its Gotenberg PDF) to the depth of a premium solar proposal — the reference is Pylon's web proposal (sample analysed: `app.getpylon.com/proposals/vZUS2zqYuM`, "13.2kW Solar system", Solar Safari Pty Ltd) — using data the deterministic engine already receives or can receive from APIs we already call. The deterministic engine and its guardrails remain the single source of truth for every dollar figure; nothing in this feature lets an external service or a generative model override a grounded number.

### Reference proposal anatomy (extracted from the live Pylon sample)

Cover → Proposed panel layout (aerial + panel rectangles + trust badges) → Panel strings & component markings (aerial + colored string runs) → System details (daily-production-per-month chart, component cards) → Utility costs (before solar / with solar) → Included services & warranty → 20-year financial summary (NPV, discounted payback, total ROI, IRR) → Financial analysis (annual bill over time, monthly bill comparison) → Environmental analysis → Quote table + payment → Assumed values (DC array power, tilt, azimuth) → Assumptions & disclaimer → T&Cs → Warranty + manufacturers tables → System summary.

## 2. Current state (audited 2026-06-12)

### Google Solar API — consumed today

`maxArrayPanelsCount`, `panelCapacityWatts`, `solarPanelConfigs[]` (count + yearly DC energy), `roofSegmentStats[]` pitch/azimuth/area (via the shared roofing parser), `imageryQuality`, `imageryDate`; `dataLayers` is summarised to presence booleans only (`lib/solar/data-layers.ts` — GeoTIFF URLs deliberately not persisted).

### Google Solar API — available but unused (the gap this spec closes or defers)

| Field | Unlocks | This spec |
| --- | --- | --- |
| `solarPotential.solarPanels[]` (per-panel center, orientation, segmentIndex, yearly energy) | Engineering-accurate panel layout + string runs | **Build** |
| `carbonOffsetFactorKgPerMwh` | Environmental analysis section | **Build** |
| `roofSegmentStats[].panelsCount` | Per-plane panel distribution | **Build** (feeds overlays) |
| `wholeRoofStats` | Cross-check of our area sums | **Build** (validation only) |
| `financialAnalyses[]` | Google's own ROI model as a benchmark | Defer — we keep our deterministic economics |
| dataLayers GeoTIFF rasters (annual/monthly flux, hourly shade, DSM/RGB/mask) | Irradiance heatmaps, hour-by-hour shade | Defer (future phase; large rasters, decode cost) |

### Image pipeline today

Hero = real Google Static Maps satellite photo (`/api/solar/q/[token]/static-map`), stats overlay from `hero-overlay.ts`. "Panels-after" = Gemini image-to-image (`panels-after.ts`), placement **illustrative** (prompt-driven, not geometry-driven), cached at `solar_estimates.panels_image_path`.

### Other relevant facts

- No customer energy/bill input exists anywhere in the solar flow; economics uses config defaults (`self_consumption_pct` 0.40, fixed retail rate).
- Quote PDF parity shipped 2026-06-12 (migration 106: `solar_estimates.pdf_path`, `lib/solar/report-html.ts`, `/api/q/solar/[token]/pdf`, confirm-time customer SMS + best-effort MMS).
- Confirm gate: prices render only after `confirmed_at` (`resolveSolarQuoteView`); inspection-routed estimates are price-free.

## 3. Pylon API findings (docs analysed 2026-06-12)

`api.getpylon.com` is REST/JSON:API, Bearer token (server-side only; per Pylon's own docs a token shared via chat must be revoked — **the key pasted in chat history must be rotated and stored only as `PYLON_API_KEY` in `.env.local`/Vercel env**). Billing per their pricing page.

**Decisive constraint: solar designs and proposals cannot be created via the API.** Designs are made by a human in Pylon's studio; the API only reads them back (`GET /v1/solar_designs`, `GET /v1/solar_designs/{id}` — exposing `summary.latest_snapshot_url`, `summary.single_line_diagram_pdf_url`, `summary.pv_site_information_url` (AS/NZS 5033), `summary.web/pdf_proposal_url`, full `line_items[]` with AU GST/STC tax semantics, `pricing`). Therefore an instant QuoteMate estimate can never be "generated by Pylon" — premium visuals must be built in-house.

**Endpoints adopted in this spec (light integration):**

| Endpoint | Use |
| --- | --- |
| `GET /v1/au/stc_amount` (no permissions required) | Official STC quantity/zone calculator — cross-check guardrail for our deterministic STC math. Returns `stcs`, `zone`, `zone_rating`, `deeming_period`. |
| `POST /v1/opportunities_form` | Optional CRM lead push: a confirmed QuoteMate solar estimate becomes a lead in the tenant's Pylon pipeline. |

**Deferred to a future phase (out of scope here):** Pylon design-import (pull `latest_snapshot_url`, single-line diagram and line items into the QuoteMate quote for tenants who design in Pylon), component-datasheet enrichment (`/v1/solar_modules|solar_inverters|solar_batteries/{sku}` — only useful with SKUs, which come from designs), files/events/pipelines.

## 4. Design

### 4.1 Data layer — engine extension (`lib/solar/roof.ts`, `types.ts`, `intake.ts`)

- Parse and persist from `buildingInsights.solarPotential`:
  - `solarPanels[]` → `SolarRoofFacts.panels: SolarPanelPlacement[]` — `{ center: LatLng, orientation: 'LANDSCAPE'|'PORTRAIT', segment_index: number, yearly_energy_dc_kwh: number }`, capped/truncated to the largest tier's panel count at render time (the array itself stores up to `maxArrayPanelsCount` entries).
  - `panelHeightMeters` / `panelWidthMeters` → `SolarRoofFacts.panel_size_m` (drives drawn rectangle dimensions).
  - `roofSegmentStats[].panelsCount` → per-plane counts on `SolarRoofPlane`.
  - `carbonOffsetFactorKgPerMwh` → `SolarRoofFacts.carbon_offset_factor_kg_per_mwh: number | null`.
  - `wholeRoofStats.areaMeters2` → logged cross-check against our summed plane area; divergence >15% appends guardrail flag `roof_area_mismatch` (review-forcing, non-blocking).
- **Optional bill input:** `SolarEstimateRequestSchema` gains `energy: { quarterly_bill_aud?: number }` (positive, ≤ 10 000); form gains one optional field ("Quarterly power bill · optional — personalises your savings"); value flows through `form-payload.ts` → route → `estimate.context.quarterly_bill_aud`. When present, economics derives `household_annual_kwh = (bill × 4) / retail_rate` and computes personal before/after utility costs; absent → current config-default behaviour, charts labelled "modelled on typical usage".
- All new data rides inside the existing `solar_estimates.estimate` jsonb — **no DB migration**. Manual-fallback estimates have `panels: []` (no geometry exists); every consumer must tolerate that.

### 4.2 Visual layer — pure SVG builders (shared page + PDF)

All three modules are pure functions (no I/O) returning SVG strings, in `lib/solar/`, unit-tested. The page embeds them inline; `report-html.ts` embeds the same strings in the PDF; both render over the existing static-map image (same Mercator projection the static map uses, centred via `static-map-center.ts` logic).

- `layout-overlay.ts` — projects each `panels[i].center` to pixel space at the static map's zoom/size; draws panel rectangles (using `panel_size_m` scaled by latitude-corrected metres-per-pixel, rotated per plane azimuth), colour-keyed per `segment_index`. Output: the "Proposed panel layout" figure. Caption carries the existing imagery-date line.
- `string-overlay.ts` — chunks the headline tier's panels into indicative strings: group by `segment_index`, order along the plane, split into runs of `min(panels_per_plane, max_string_len)` where `max_string_len` defaults to 14 (config: `string_max_panels`). Distinct colour + numbered terminal markers per string; inverter/meter marker at the polygon edge nearest the street-facing vertex. Output: "Panel strings & component markings", always captioned **"Indicative string layout — final stringing is confirmed by your installer at site"**.
- `charts.ts` — four pure SVG charts: monthly production bars (seasonal AU curve scaled to `annual_kwh_ac`, labelled modelled — monthly flux rasters are deferred), utility costs before/with-solar bars, monthly bill comparison, 25-year cumulative savings line (degradation applied). Tabular figures, axes, AU currency formatting; print-safe colours matching the existing report theme.
- Gemini `panels-after.ts` is unchanged in role (photorealistic hero, labelled illustrative); its prompt additionally receives the real per-plane panel counts so the render tracks the actual distribution.

### 4.3 Financial + environmental math (`lib/solar/financial-summary.ts`)

Pure, per tier, from existing `SolarEconomicsTier` + new config params (`solar_config`: `price_escalation_pct_per_year` default 0.03, `discount_rate_pct` default 0.05 — config-versioned like every other constant):

- Year-by-year 20-year projection: savings(y) = year-1 savings × (1 − degradation)^y × (1 + escalation)^y.
- Outputs: cumulative savings series (feeds the chart), **NPV** (discounted at `discount_rate_pct`), **payback band** (existing low/high), **total ROI %** (cumulative ÷ net price), **IRR** (bisection on the cashflow series; null when not bracketable).
- Environmental: `annual_kwh_ac × carbon_offset_factor_kg_per_mwh / 1000` → tonnes CO₂e/yr + 20-year total; equivalents (trees, km driven) via cited constants in config. Section omitted when the factor is null (manual fallback).
- Compliance: every financial figure carries the existing confidence-band framing; new copy in `compliance-copy.ts`: modelled projections, not financial advice; assumptions table lists escalation/discount/degradation rates explicitly.

### 4.4 Quote restructure — `/q/solar/[token]` page + PDF (same order)

1. Hero (existing satellite + stats + Gemini after-image)
2. Proposed panel layout (layout-overlay)
3. Panel strings & component markings (string-overlay)
4. System details — tier cards (existing) + monthly production chart + assumed-values table (DC kW, panel count, tilt, azimuth, derate, config version)
5. Utility costs — before/with-solar (personal when bill given; modelled otherwise, labelled)
6. 20-year financial summary — NPV / payback band / ROI / IRR stat cards
7. Financial analysis — bill-over-time + monthly comparison charts
8. Environmental analysis
9. Pricing & acceptance — existing G/B/B tiers, STC subtraction lines, deposit CTA (Stripe), unchanged
10. Assumptions & disclaimers (extended compliance copy)

Gating: sections 5–7 and 9 (anything with $ or savings) stay behind `confirmed_at` exactly as today; sections 2–4 and 8 (geometry, production, CO₂ — no dollar figures) may render pre-confirm. Inspection-routed estimates keep the current price-free page. `report-html.ts` mirrors the same section order with the same SVG strings; PDF regenerates on confirm (existing `ensureSolarQuotePdf` flow).

### 4.5 Pylon light integration (`lib/pylon/client.ts`)

- Server-only client: `PYLON_API_KEY` from env, `Authorization: Bearer`, JSON:API accept headers, 5 s `AbortSignal.timeout`, never throws to callers (result objects).
- **STC cross-check** — in the estimate route's `after()` (and on re-draft): for each priced tier call `GET /v1/au/stc_amount?sgu_kind=solar_deemed&output_kw=…&site_postcode=…&installation_year=…`; compare `stcs` to our `SolarStcBreakdown.certificates`. `|Δ| > 1` certificate ⇒ append guardrail flag `stc_mismatch_pylon:{tier}` (forces tradie review via the existing flag gate — flagged estimates cannot be confirmed until re-drafted clean). Result cached on the estimate (`context.pylon_stc_check`) for display in the assumed-values table ("STC count verified against Pylon ✓").
- **CRM lead push** — on first confirm, when tenant flag `pylon_lead_push` is on: `POST /v1/opportunities_form` with name/phone/address/system summary. Fire-and-forget in `after()`, logged, never blocks confirm.
- Flags: `PYLON_ENABLED` env gate (default off) + per-tenant `pylon_lead_push` (in `tenants` branding/config jsonb, default off). Pylon unreachable ⇒ skip silently with a log line; the estimate flow is bit-identical to today.

### 4.6 Error handling & degradation matrix

| Condition | Behaviour |
| --- | --- |
| `panels[]` empty (manual fallback / old estimate) | Layout + strings sections omitted; page/PDF render as today |
| Bill not provided | Utility/financial charts use config defaults, labelled "modelled on typical usage" |
| `carbon_offset_factor` null | Environmental section omitted |
| Pylon down / key missing / `PYLON_ENABLED` off | No cross-check, no flag, no lead push; log only |
| Gemini render fails | Existing fallback (plain satellite) — unchanged |
| Gotenberg unset | Existing behaviour (no PDF link/MMS) — unchanged; health probe reports it |

### 4.7 Testing

- Unit (vitest, colocated): geometry projection (known lat/lng → pixel), rectangle rotation, string chunking (plane grouping, max length, ordering), all four chart builders (snapshot + numeric label assertions), financial-summary math against hand-computed cases (incl. IRR bracketing edge cases), STC-comparison flag logic, Pylon client error paths (mocked fetch), schema acceptance of the optional `energy` block, payload builder omission semantics.
- Existing suites must stay green: `tier-cards`, `economics`, `pricing`, `quote-page-row`, `report-html` (extended), `estimate-route`.
- E2E smoke: `/q/solar/[token]` renders all sections for a google-path fixture and degrades for a manual-path fixture.

## 5. Build order (phases, each shippable behind `SOLAR_PREMIUM_QUOTE`)

1. **Data** — parse/persist new Solar API fields + optional bill input (schema → form → route → economics personalisation).
2. **Visuals** — layout-overlay, string-overlay, charts; embed in page + PDF behind the flag.
3. **Financials & restructure** — financial-summary, environmental, compliance copy, full section reorder of page + PDF.
4. **Pylon** — client, STC cross-check guardrail, tenant-flagged lead push. Rotate the exposed key before this phase ships.

## 6. Out of scope (recorded for the next iteration)

Pylon design-import (snapshot/single-line diagram/line items for Pylon-studio tenants), GeoTIFF flux heatmaps + hourly-shade visuals, Google `financialAnalyses` benchmark display, battery/EV-charger add-on tiers (Pylon component catalogs), financing products, e-signature. None of these block the premium quote; each layers on top of the data captured here.
