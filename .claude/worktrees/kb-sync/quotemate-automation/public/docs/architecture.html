QuoteMate Automation Pipeline · Stages 01 → 10 · Voice + SMS

From inbound contact to booked job. Two channels (voice + SMS), one shared intake/estimation pipeline, end-to-end shipped in production.

Twilio AU Vapi Deepgram nova-2 ElevenLabs Claude Haiku 4.5 · Opus 4.7 Voyage voyage-3 + Rerank-2.5 Vercel · Next.js 16 Supabase + pgvector Vercel AI SDK Stripe Checkout + Connect WhatsApp (WABA)
v1 shipped
Production status as of 2026-05-06. Both channels are live. Voice via Vapi (Jeff) on the Twilio AU number; SMS via Twilio direct → Haiku 4.5 dialog agent on +61 481 613 464. Stages 04–07 (intake structuring, embedding, RAG+rerank, estimation) are identical for both channels — the routes accept a discriminated-union body and branch only on which source table to read. Stages 08–10 (Stripe Checkout, customer SMS dispatch, HTML quote page, booking, tradie notify) are likewise channel-agnostic. The diagrams below show the voice path; an "SMS parallel" section follows that explains where SMS forks and where it rejoins.
01 · The whole pipeline at a glance

End-to-end data flow.

Top to bottom: a homeowner dials in, the AI has a conversation, the transcript gets structured, and an LLM with deterministic tools writes a draft quote. Every box is a single piece of infrastructure or one Vercel AI SDK call. Every arrow is one HTTP request.

02 · Stage-by-stage breakdown

What runs at each step.

Each card is one stage from the wireframe. Read the diagram above for the wiring; read these cards for what each stage actually does and which tool does the work.

Stage · 01

Customer call origination

no infrastructure

A homeowner reaches for their phone and dials the tradie's QuoteMate-provisioned number. Same number every day, no after-hours fallback, no IVR menu.

tools
  • customer's mobile / landline
Stage · 02

Twilio AU number receives

~$1.50/mo + per-minute · ~$0.08/call

Twilio owns the AU local long code. On purchase, its voice webhook is delegated to Vapi via the Vapi dashboard's "Import from Twilio" action — Twilio terminates SIP, hands the audio stream to Vapi.

tools
  • Twilio Console (regulatory bundle)
  • Twilio AU local number (voice cap.)
why local, not 1300
  • homeowners answer local calls; 1300s feel like marketing
Stage · 03

Vapi runs the conversation

~10Hz loop · ~$0.30–0.55/call

Vapi orchestrates the realtime audio loop: Deepgram transcribes incoming audio, Claude Haiku reads the transcript and the system prompt to decide what to say next, ElevenLabs synthesizes the reply. ~10 cycles per second keeps conversation natural.

On hangup, Vapi POSTs an end-of-call-report with the full transcript + recording URL to /api/vapi/webhook.

tools
  • Vapi (orchestrator)
  • Deepgram nova-2 (en-AU)
  • Claude Haiku (in-call)
  • ElevenLabs (Australian voice)
  • Vercel API route (webhook receiver)
Stage · 04

Intake Engine structures

Claude Opus 4.7 + vision · ~$0.04/call

Free-form transcript → structured JSON. The Vercel AI SDK's generateObject enforces a Zod IntakeSchema — Opus must produce JSON matching it or the SDK retries. Customer-supplied photos (MMS attachments, upload-link uploads, in-call photo SMS uploads) pass through as image content blocks for vision grounding. Channel-agnostic: same handler reads {callId} (voice) or {conversationId, sourceChannel:'sms'}.

A summary string ("downlights count=6 indoor existing-wiring...") is embedded with Voyage voyage-3 into a 1536-dim vector and stored in pgvector for similar-job retrieval. A quality gate fires if name/scope/job_type are missing → empty intake gets a callback-request SMS, no quote drafts.

tools
  • Vercel AI SDK generateObject
  • Zod schema validation
  • Anthropic (Opus 4.7 + vision)
  • Voyage AI (voyage-3 embeddings)
  • Supabase pgvector extension
  • evaluateIntakeQuality gate
Stage · 05

Estimation Engine drafts

Claude Opus 4.7 + RAG + Rerank · ~$0.05–0.10/call

generateText({ tools, maxSteps: 10 }). Before the LLM call, the RAG context builder pulls similar past intakes via match_intakes() (job_type pre-filter → pgvector top-20) and Voyage Rerank-2.5 cross-encodes them down to the top 3, then formats a "here's how you've quoted similar jobs" block into the Opus prompt. This anchors pricing to actual past quotes instead of free-form generation.

Opus then chooses which property-aware tools to call: lookup_assembly, lookup_material, apply_markup, flag_inspection_needed. Tools hit Supabase deterministically; Opus only handles reasoning. After generation a semantic-grounding validator checks every line item's description matches its source row — ungrounded items auto-downgrade to the $99 inspection route. Output is three pricing tiers (Good/Better/Best) as JSONB columns. Status starts as 'draft'.

tools
  • Vercel AI SDK generateText + tools
  • RAG context builder (lib/estimate/rag.ts)
  • Voyage Rerank-2.5 cross-encoder
  • 4 property-aware Zod-typed tools
  • Semantic-grounding validator (4th defence layer)
  • Anthropic Opus 4.7
  • Supabase lookup tables (read)
  • Supabase quotes (write JSONB)
Stage · 06

Stripe Checkout + dispatch

Stripe Connect Express + Twilio · ~$0.01/quote

One Stripe Checkout Session is created per tier (Good/Better/Best) for the deposit (default 30%), each linked through a redirect at /r/[token]/[tier] so expired sessions can be regenerated lazily. For inspection-required quotes, one $99 site-visit Session is created instead.

Then the customer-facing quote SMS goes out via the shared dispatchQuoteMessage (SMS-first, WhatsApp-fallback). For SMS-sourced quotes, the dispatch uses TWILIO_SMS_NUMBER as the from-address so the quote lands in the same thread as the dialog.

tools
  • Stripe Checkout Sessions API
  • Twilio REST (SMS + WhatsApp)
  • buildQuoteSms · dispatchQuoteMessage
  • Per-tier /r/[token]/[tier] redirect
Stage · 07

HTML quote page

Next.js 16 RSC · /q/[token] · force-dynamic

Customer taps "View full quote" → mobile-friendly HTML page renders Good/Better/Best with mocked tier photos, scope of works, assumptions, risk flags, timeframe, GST note, NECA NSW licence footer. Per-tier "Lock in this option · $X deposit" buttons → Stripe Checkout via the redirect.

Customer-uploaded photos render via a render-time union of intakes.photo_paths (snapshot at intake time) AND the live source-of-truth table (calls.photo_paths for voice or sms_conversations.photo_paths for SMS) — late uploads appear on refresh without a re-quote. Signed URLs regenerate on every render (24h expiry).

tools
  • Next.js 16 RSC (server components)
  • Tailwind v4 styling
  • refreshSignedUrl on demand
  • getTierPhoto mock images
Stage · 08

Stripe webhook + paid page

Stripe webhook · /api/stripe/webhook

Customer pays the deposit → Stripe fires checkout.session.completed → webhook sets quotes.paid_at + paid_tier. Stripe redirects the customer to /q/[token]/paid which shows confirmation and a "Pick a time" CTA into the booking flow.

tools
  • Stripe webhooks (signature-verified)
  • /q/[token]/paid · /q/[token]/cancelled
  • Supabase quotes (write paid_at, paid_tier)
Stage · 09

Booking — slot picker

Next.js 16 RSC · /q/[token]/book

Customer chooses an available time slot (slots seeded per-tradie via scripts/seed-tradie-slots.mjs). Locking a slot fires POST /api/q/[token]/book which marks the slot taken and links it to the quote.

tools
  • Next.js 16 RSC + client SlotPicker
  • Supabase tradie_slots + bookings
Stage · 10

Booking confirmations + tradie notify

Twilio (SMS + WhatsApp) · ~$0.02/booking

On successful slot lock, two messages fire in parallel: customer gets buildBookingConfirmationSms ("you're locked in for Thu 7 May, 9:00am"); tradie gets buildTradieBookingNotification via SMS-first/WhatsApp-fallback to TRADIE_NOTIFY_NUMBER + WhatsApp to TRADIE_NOTIFY_WHATSAPP.

Separate from booking, the tradie also gets a notification on every SMS-sourced quote draft (buildTradieDraftNotification) so they see new leads in real time.

tools

  • dispatchQuoteMessage shared dispatcher

02.5 · SMS channel — where it forks, where it rejoins

The same pipeline reachable via SMS instead of voice.

SMS replaces only Stages 02–03. From Stage 04 onward (intake → embed → RAG → estimate → Stripe → quote → booking) every step is identical to voice. The fork happens at the Twilio webhook; the rejoin happens at /api/intake/structure.

SMS · 02

Twilio webhook (SMS direct)

No Vapi · /api/sms/inbound

Customer texts +61 481 613 464 → Twilio POSTs straight to our route (no Vapi involved for SMS). Signature-validated against TWILIO_AUTH_TOKEN. Idempotency check on MessageSid (route + DB unique index, migration 004) prevents duplicate replies on retry.

Inbound row persisted in sms_messages; route returns 200 in <500ms (fast-ack via after()) so Twilio never times out.

SMS · 03

Haiku 4.5 dialog turn

Anthropic Haiku 4.5 · ~$0.001/turn

One generateObject call per inbound. Returns Zod-validated {action: 'ask' | 'finish' | 'escalate_inspection'} with reply text capped at 320 chars. 8-step decision guide: inspection trigger → off-topic redirect → no job_type → out-of-scope job → name → suburb → per-job mustAsk → finish. Capped at 4 turns.

First turn includes greeting + gratitude ("G'day, thanks for messaging QuoteMate — I'm the AI quoting assistant…"); follow-up turns drop the intro and stay direct. On finish, fires the channel-agnostic intake handler with {conversationId, sourceChannel:'sms'}.

SMS · ↩

Rejoin point: Stage 04 onward

Channel-agnostic · same pipeline as voice

/api/intake/structure accepts a discriminated-union body. SMS branch loads sms_conversations + sms_messages, stitches a transcript, prepends accumulated assumptions, aggregates photos from MMS + upload-link surfaces. Everything downstream — embed, RAG, rerank, estimate, Stripe, dispatch, booking — is identical.

02.6 · Photo flow — cross-cutting

How customer photos enter the pipeline (from either channel).

Two surfaces feed the same intake. Vision sees the snapshot at intake time; the customer-facing quote page sees the live union (so late uploads still appear on refresh).

Photo · 01

Photo-request SMS dispatched

Voice path: Vapi tool send_sms_photo_link fires mid-call. SMS path: dialog route fires when easy-5 job_type is identified on turn 1 (single-fire enforced via photo_request_sent_at). Both produce ${'${APP_URL}'}/upload/${'${token}'} via buildPhotoRequestSms.

Photo · 02

Two upload surfaces

MMS: customer attaches photos to the SMS thread → lib/sms/mms.ts fetches each MediaUrlN (Basic auth) → uploads via uploadIntakePhoto → sms_messages.photo_paths. Upload link: customer taps /upload/[token] → camera/gallery picker → /api/upload/[token] resolves token to either calls.photo_paths or sms_conversations.photo_paths based on which table owns the token.

Photo · 03

Snapshot vs live (race-safe)

Snapshot: /api/intake/structure aggregates both surfaces into intakes.photo_paths at run time — that's what Opus vision sees. Live union: /q/[token] reads intakes.photo_paths ∪ live source table on every render and re-signs URLs (24h TTL). Late uploads appear on refresh — vision price stays consistent because the snapshot froze at intake time.

02.7 · Production hardening

Reliability layers added beyond the basic flow.

Idempotency, fast-ack, signature validation, race protection, and graceful degradation. Most are invisible when things go right; they're the things that stop you waking up to a bug-bash.

Hardening · 01

Inbound webhook idempotency

Three layers: (1) application-layer SELECT-then-skip on twilio_message_sid; (2) DB-level partial unique index (migration 004) catches concurrent retry races with 23505 error → 200 ack; (3) fast-ack via after() so Twilio never times out and never retries.

Hardening · 02

Twilio signature validation

lib/sms/twilio-validator.ts verifies every inbound webhook against TWILIO_AUTH_TOKEN. Reconstructs the URL from forwarded-host headers so signatures match even when Vercel internal routing rewrites the URL. Forged requests get 403.

Hardening · 03

Strict-grounding defence layers

(1) System prompt forbids invented prices; (2) all $ come from tool results, never free-form; (3) RAG anchors to past quotes; (4) post-generation semantic-grounding validator checks line description ↔ price source. Any ungrounded line auto-downgrades to $99 inspection.

Hardening · 04

Daily cleanup cron

/api/cron/sms-cleanup runs daily at 17:00 UTC (vercel.json). Sweeps sms_conversations status='open' AND last_message_at < now()-24h → marks them 'abandoned'. Bearer-secured via CRON_SECRET.

Hardening · 05

WhatsApp fallback dispatcher

dispatchQuoteMessage tries SMS first; on carrier-reject codes (21612, 21408, etc.) falls back to WhatsApp via the registered WABA sender (+1 555 941 5397, "NomanuAI"). Same dispatcher used by voice + SMS + tradie notify + booking confirmation.

Hardening · 06

Stripe Session re-issuance

Per-tier redirect (/r/[token]/[tier]) regenerates a fresh Stripe Checkout Session if the original expired (24h TTL). Customer never lands on a broken pay link.

03.5 · SMS + booking tables (added since the diagram above)

Tables not shown in the SVG schema.

The diagram above shows the original voice-path tables. Below are the tables added for the SMS channel, the photo flow, and the booking flow — all FK-linked into the same intakes + quotes rows so the rest of the pipeline doesn't care which channel the data came from.

Table · sms_conversations

One row per ongoing dialog

migration 002 · extended by 003, 005

Tracks the dialog state for an SMS thread. status ∈ open / structuring / done / abandoned. intake_id FK back to intakes set when dialog finishes. photo_request_token + photo_urls + photo_paths for upload-link photos.

columns
  • id · from_number · to_number · status
  • turn_count · assumptions_made (jsonb)
  • intake_id (FK)
  • photo_request_token · photo_request_sent_at
  • photo_urls · photo_paths · photos_completed_at
  • created_at · updated_at · last_message_at
Table · sms_messages

One row per inbound or outbound

migration 002 · extended by 003, 004

Append-only log of every message in the conversation. Inbound rows have twilio_message_sid with a partial unique index (migration 004) preventing duplicate persists from webhook retries. photo_urls + photo_paths hold inbound MMS attachments.

columns
  • id · conversation_id (FK) · direction
  • body · twilio_message_sid (unique inbound)
  • photo_urls (jsonb) · photo_paths (jsonb)
  • created_at
Table · tradie_slots

Bookable time slots

f3 booking schema

Pre-seeded availability windows per tradie. Read by /q/[token]/book. Slots get marked is_taken=true when a customer locks one in.

columns
  • id · tradie_id · scheduled_at
  • duration_minutes · is_taken
  • created_at
Table · bookings

Confirmed jobs

f3 booking schema

One row per locked-in slot. FK to the quotes row and the tradie_slots row chosen. Snapshots scheduled_at directly so the booking remains valid even if the underlying slot row is later edited.

columns
  • id · quote_id (FK) · slot_id (FK)
  • scheduled_at · status
  • created_at
Extension · intakes

Photos + photo paths

The intakes row gained photo_paths (permanent storage paths) alongside photo_urls (24h signed URLs). Paths are persistent; URLs get re-signed on demand at /q/[token] render time. call_id is null for SMS-sourced intakes (the discriminator).

Extension · quotes

Stripe + sharing

Quotes gained share_token (unguessable URL slug for /q/[token]), stripe_links (per-tier session URLs as JSONB), paid_at + paid_tier (set by the Stripe webhook), routing_decision (HIGH/MED/LOW from confidence routing), and scope_short + selected_tier for the customer SMS body builder.

04 · Why no n8n (or any workflow tool)

Three architectural reasons.

n8n is an excellent SaaS-glue tool — it shines for "when row added in Sheet → post to Slack → create Trello card." It's the wrong shape for this pipeline. Here's why each of the three core stages defeats it.

REASON · 01

Stage 03 is sub-second realtime.

Vapi's audio loop runs ~10 cycles per second: audio chunk → Deepgram → Haiku → ElevenLabs → audio chunk. n8n is event-batched HTTP — it can't be in a streaming voice loop. Vapi already is the orchestrator for this stage; bolting n8n on just intercepts the post-call webhook for no benefit.

streaming · sub-second turn-taking
REASON · 02

Stage 04 needs schema-enforced output.

The Intake Engine uses generateObject({ schema: IntakeSchema }) — Sonnet must produce JSON matching a Zod schema, with automatic retries if it doesn't. n8n's HTTP-call-and-parse can't enforce that. The moment you reach for a "code node" inside n8n to do it, you're writing the same code — just inside a slower, vendored, harder-to-test environment.

Zod · generateObject · retry-on-mismatch
REASON · 03

Stage 05 needs LLM tool agency.

generateText({ tools, stopWhen: stepCountIs(10) }) lets Opus choose which of four tools to call, in which order, based on what the prior tool returned. n8n can chain HTTP calls in a fixed graph, but it cannot give the LLM agency to pick the next tool mid-loop. That agency is the whole architectural point — money-touching steps must be tool-calls, never free text.

tool-use loop · stopWhen · LLM-decided routing
where n8n CAN fit
n8n is a fine choice for the post-quote glue work — Stages 06+: notify the tradie on Slack when a quote is ready, sync accepted quotes to Xero/MYOB, send follow-up SMS sequences, calendar bookings. Those are CRUD between SaaS apps and exactly n8n's strength. Stages 02–05 are not.
